Ti-Project-Assistant使用教程

仓库地址：Ti-Project-Assistant

1.前言

这篇文章是一个关于我的开源小项目，也就是这个ti-project-assistant的使用教程，主要是帮大家更快地开始M0芯片的开发，主要是面向电赛的同学们，无论你是新手还是老手，这个项目可以更快地帮你上手业务开发，并且按照最前沿、最现代的方式组织自己的项目，通过CMake和VSCode的结合，实现完全摆脱Keil和CCS Theia的束缚，使用现代化的开发方式进行全流程配置、开发、编译、烧录和调试。

我的项目已经开源在Github，为了满足国内网络环境的需求，后续会迁移到Gitee上，大家可以通过网站获取完整开源项目，也可以直接从这个博客上下载脚本源码。

2.项目介绍

这个项目用Python编写，通过环境变量配置读取Sysconfig（TI官方图形化芯片配置工具）和M0SDK（TI官方提供的M0芯片开发套件）的位置，通过读取syscfg配置文件，自动调用代码生成脚本sysconfig cli，在生成必要配置代码的基础上，简单封装项目结构、编写CMakelists文件，通过符合现代工程实践的目录架构组织项目，大家可以直接在VSCode中打开项目，然后再主程序的位置开始项目业务逻辑的开发或是底层驱动的编写。该脚本还会生成适用于VSCode的launch.json和tasks.json文件，通过VSCode的TI官方插件提供的调试器和官方OpenOCD，方便大家直接在VSCode中进行编译、烧录和调试。

这个项目的设计初衷是为了方便自己更轻松地配置项目结构，实现全流程Linux环境下的嵌入式开发。在完成后进行了简单的封装和Windows的适配，变成了现在这个可以供大家学习参考的开源工具，希望能在电赛中帮上大家的忙。

也就是说，你只需要安装最基本的开发依赖、配置芯片、运行脚本的“三步走”流程，就可以快速进入开发，直接享用VSCode带来的现代化开发体验。读者可以自行接入Claude或是Copilot等AI插件，实现全流程的智能开发体验。

3.使用教程

用前须知

  • 大部分嵌入式开发者都是在Windows系统进行开发的，本项目针对Windows做了适配，所以大家是可以在Windows系统下直接使用的。

3.1 安装依赖

在 Windows 上使用这个项目，建议先把下面这些基础依赖准备好。它们分别负责脚本运行、交叉编译、构建系统、TI 官方配置生成，以及后续的下载和调试。

0）一点碎碎念

对于不是很清楚编译、烧录、调试流程的用户来说呢，可以看一下这一小节的解释，我尽量简单地讲解一下嵌入式开发流程，让你对接下来的步骤熟悉一点，至少知道自己在做些什么。清楚流程的小伙伴们可以直接进入依赖的安装。

当我们在进行嵌入式开发的时候，通常会经历几个阶段：编写代码->编译->烧录->调试。

编写代码是我们最熟悉的阶段，通常我们会在VSCode或是其他IDE中编写C/C++代码，然后保存到src目录下。

接下来我们按下编译器的编译按钮，神秘的编译器就会按照某种规则，帮我们把代码转换成机器可以阅读的二进制文件，这个过程就是编译。当然实际上这个编译过程也分成几个更细小的步骤，比如链接、预处理、汇编等等，但我们不需要关心这些细节，只要知道编译器会把我们的代码变成二进制文件就行了。无论是什么嵌入式或是一般的C/C++程序，我们都需要一个编译器来帮我们进行代码到可执行文件的转换。对于嵌入式开发来说，这个步骤体现为生成一个.elf文件，它规定了程序在芯片上的运行方式和位置，其实也就是一串0和1构成的文件。

编译器生成的二进制文件是不能直接在电脑上运行的，因为它是为芯片设计的，所以我们需要一个烧录器，把这个二进制文件传输到芯片中去。烧录器通常是一个硬件设备，它通过USB或其他接口连接电脑和芯片，然后把编译器生成的二进制文件写入芯片的闪存中。当我们按下烧录的按钮，VSCode就会调用一个烧录程序，它可以帮我们把二进制文件按照某种芯片能够读懂的顺序下载到芯片里面。在硬件上，这个就是我们的烧录器，可以是TI官方的XDS110，也可以是比如DAPLink或是JLink等。他们各自遵循自己的烧录语言，所以我们需要一个特定的电脑程序来和烧录器进行通信，这个程序就是OpenOCD，它是一个开源的烧录和调试工具，支持多种烧录器和芯片，也就是说它能够使用比如CMSIS-DAP协议和XDS110协议与烧录器进行通信，然后把二进制文件传输到芯片中去。

在代码烧录到芯片上之后，我们就可以直接观察到代码的运行结果了。但是如果想对代码进一步调试或者分析呢？这就需要我们一步一步监控代码的运行，观察代码运行到每一步时芯片的反应，比如引脚的信号或是寄存器的数据。这个过程就是调试。下载器一般可以兼具调试功能，通过和电脑连接，我们能够实时从电脑暂停芯片运行，或是直接读取芯片数据，这个过程也可以用下载器和下载程序实现。在这个环境下，下载器其实也是调试器，我们的OpenOCD程序就作为调试服务器，监听电脑下发的命令，比如说暂停芯片运行、读取寄存器数据、设置断点等等，然后把这些命令传输给下载器，下载器再把这些命令传输给芯片，芯片执行完命令后再把结果返回给下载器，下载器再把结果返回给OpenOCD，最后OpenOCD把结果返回给VSCode。VSCode上的gdb作为调试客户端，负责和OpenOCD进行通信，把用户的操作转化为调试命令发送给OpenOCD，然后接收OpenOCD返回的结果并显示在界面上。

总结一下，上面的一整个过程需要涉及到：

  • 编译器（arm-none-eabi-gcc）：把代码编译成芯片可执行的二进制文件。
  • 烧录器（DAPLink）：把二进制文件下载到芯片中。
  • 烧录程序（OpenOCD）：电脑端的程序，负责和烧录器通信，把二进制文件传输到芯片中。
  • 调试客户端（arm-none-eabi-gdb）：下载器也可以作为调试器，负责和电脑通信，执行调试命令。
  • 调试服务器（OpenOCD）：电脑端的程序，负责和调试器通信，把用户的调试命令传输给调试器，并接收调试器返回的结果。

实际上在代码的编写阶段，我们的整个项目的布局也是很重要的，如何编写出一个易于开发、易于拓展、易于迁移的项目是我们必须考虑的问题，这不仅有利于我们自身的项目式思维的训练，对我们的开发效率也有很大帮助，高复用性也是我们能够快速开发电赛项目的基础。

为了管理项目结构和编译过程，我们使用CMake作为构建系统，它可以帮助我们自动化管理项目的编译、链接和生成可执行文件的过程。通过CMake，我们可以定义项目的目录结构、源文件、头文件和依赖库，然后CMake会根据这些定义生成适合我们平台的构建文件，比如Makefile或Ninja文件，从而简化了编译过程。

简单来说就是用一个特殊的工具，CMake，帮我们自动链接头文件和源文件，配置运行参数等。构建器比如ninja或是make，就通过CMake生成的构建文件（运行指南）去帮我们调用编译器和链接器，最终生成可执行文件。

2）ARM GCC 交叉编译工具链

这是用来编译 MSPM0 工程的编译器，建议安装 ARM GNU Toolchain 13.x，这个作为硬编码进脚本的默认编译器，暂时不支持其他编译器，后续会考虑支持更多编译器。
对于从来没有安装过C/C++开发环境的用户来说，可能需要考虑安装MSys2或是完整的MinGW环境，安装包括ninja、CMake、arm-none-eabi-gcc在内的完整工具链，对于有经验的用户，也可以从STM32CubeIDE或是其他嵌入式开发环境中提取arm-none-eabi-gcc工具链，放到系统PATH中即可，确保能够从终端直接找到 arm-none-eabi-gcc。

这里提供两种安装方式：

  • 推荐方式：从 ARM 官网下载安装包，安装时选择添加到系统 PATH，这样可以直接在命令行中使用 arm-none-eabi-gcc。
  • 使用Mys2或MinGW等包管理器安装，用Pacman安装 arm-none-eabi-gcc，并将其路径添加到系统 PATH 中。
    • 提醒：考虑到Windows环境需要安装较多开发工具链，推荐先去安装一下MinGW，这将是你下载和安装其他工具的基础。

CMake 负责生成工程构建文件，Ninja 负责更快地完成构建。建议两者都安装，版本要求分别保持在 CMake 3.13 以上、Ninja 任意可用版本即可。

4）TI MSPM0 SDK

这是 TI 官方提供的 MSPM0 软件开发套件，里面包含驱动库、启动文件、CMSIS 相关内容，以及项目构建所需的各种底层资源。

5）TI SysConfig

SysConfig 是 TI 官方的图形化配置工具，用来配置芯片型号、引脚复用、时钟和外设参数。脚本会调用它的 CLI 自动生成 ti_msp_dl_config.c/h 和链接脚本，所以这是必须安装的。

同样建议解压到 C:\ti\ 下，例如 C:\ti\sysconfig_1.27.1。

6）OpenOCD 和 arm-none-eabi-gdb

这两个工具主要用于烧录和调试。OpenOCD 负责和板子通信，GDB 负责源码级调试。通常它们会随 TI 的 CCS Theia 一起安装，安装完后把对应目录加入环境变量即可。
